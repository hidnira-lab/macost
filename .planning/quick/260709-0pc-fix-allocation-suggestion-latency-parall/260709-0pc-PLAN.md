# Fix allocation-suggestion latency: parallelize independent Supabase queries

## Problem

`GET /api/transactions/{id}/allocation-suggestion` measures ~2622ms average
against Railway+Supabase production (3 live runs: 2470ms, 2440ms, 2956ms),
failing the ROADMAP.md Phase 2 (02-15) success criterion of ≤2000ms.

Root cause: `backend/services/goal_service.py`'s `fetch_and_rank_goals()`
runs 4 Supabase REST queries fully sequentially, even though 2 of them have
no data dependency on the others:

1. fetch `goal` rows for user
2. fetch `alokasi` rows filtered by goal_ids from (1) — DEPENDS on (1)
3. `get_avg_monthly_side_income(user_id)` — only needs `user_id` — INDEPENDENT
4. `get_or_create_goal_settings(user_id)` — only needs `user_id` — INDEPENDENT

(1)+(2) must stay sequential relative to each other. (3) and (4) can run
concurrently with each other and with (1)+(2).

The transaction-fetch validation gate in `backend/routers/allocations.py`
(`get_allocation_suggestion()` route) must NOT be touched or parallelized —
it's a required business-rule gate that must complete before any of the
above runs.

## Confirmed facts (verified by reading source before planning)

- `backend/core/supabase.py`: `get_supabase_admin()` returns a plain
  `supabase.Client` (from the sync `supabase-py` package). No async client
  variant exists anywhere in the codebase. `.execute()` calls are blocking.
  → Parallelization must use `concurrent.futures.ThreadPoolExecutor`
  (threading), NOT `asyncio.gather` — there is nothing async to gather.
- `backend/services/goal_service.py` lines 75-102 contain the 4-query
  sequence described above, all inside `fetch_and_rank_goals()`.
- `backend/services/allocation_service.py` calls
  `goal_service.fetch_and_rank_goals(user_id)` and does no other Supabase
  I/O itself — no changes needed there.
- `backend/services/goal_settings_service.py`'s `get_or_create_goal_settings`
  is a plain sync function — safe to call from a worker thread as-is (uses
  only its own local `supabase` client reference, no shared mutable state).
- `backend/routers/allocations.py`'s `get_allocation_suggestion()` route
  fetches the transaction row first (IDOR + business-rule gate), THEN calls
  `allocation_service.get_allocation_suggestion(transaction, current_user_id)`
  — this ordering is untouched by this fix.
- API_CONTRACT.md §7 response shape for this endpoint is unchanged by this
  fix (no fields added/removed/renamed).
- Existing test `backend/tests/test_goals.py` uses a `_CountingSupabaseClient`
  wrapper that asserts total `.table()` call count (4, "never N+1") — this
  count-based assertion is thread-safe to keep passing since Python's GIL
  serializes the simple integer increment in `.table()`, and parallelizing
  execution doesn't change how many times `.table()` is called, only when.

## Task 1: Parallelize independent queries in `fetch_and_rank_goals()`

**File:** `backend/services/goal_service.py`

**Change:**
1. Add `from concurrent.futures import ThreadPoolExecutor` to the imports.
2. In `fetch_and_rank_goals()`, keep the goals fetch + `if not goals: return []`
   guard, and the alokasi fetch + `sums` aggregation exactly as-is and
   sequential (alokasi query depends on `goal_ids` from the goals query).
3. Replace the two sequential calls:
   ```python
   avg_monthly_side_income = get_avg_monthly_side_income(user_id)
   settings = get_or_create_goal_settings(user_id)
   ```
   with a `ThreadPoolExecutor(max_workers=2)` that submits both callables
   and collects results via `.result()`, e.g.:
   ```python
   with ThreadPoolExecutor(max_workers=2) as executor:
       income_future = executor.submit(get_avg_monthly_side_income, user_id)
       settings_future = executor.submit(get_or_create_goal_settings, user_id)
       avg_monthly_side_income = income_future.result()
       settings = settings_future.result()
   ```
   This must run concurrently with (and not block on) nothing else changing
   — the goals+alokasi fetch above it stays sequential and unchanged; only
   these two independent calls move into the thread pool.
4. Do not change `get_avg_monthly_side_income()`'s or
   `get_or_create_goal_settings()`'s internals, signatures, or the
   `saw_engine.rank_goals(...)` call at the end of the function — SAW logic
   is out of scope per task constraints.
5. Preserve existing exception behavior: if either future raises, `.result()`
   re-raises that exception in the main thread (same as today's sequential
   behavior would surface it) — do not add try/except that would swallow or
   alter error propagation, since that could change existing error-path test
   expectations.

**Why this is safe:** `get_avg_monthly_side_income` and
`get_or_create_goal_settings` each open/use their own Supabase client call
chain (`get_supabase_admin()` returns a cached module-level singleton, but
`supabase-py`'s `Client.table(...).select(...)...execute()` builds a fresh
request object per call — no shared mutable query-builder state between the
two functions). Both are pure read-then-maybe-write functions with no
shared local state and no ordering dependency on each other.

**Verify:**
```bash
python -m pytest backend/tests/ -k "goal or allocation" -v
```
Run from the repo root (not bare `pytest` — see `backend/pytest.ini`
rootdir/import-path quirk noted in 02-14-SUMMARY.md). All tests in
`test_goals.py`, `test_goal_settings.py`, `test_allocations.py`, and
`test_allocation_service.py` must still pass, including the
`_CountingSupabaseClient` query-count assertion in `test_goals.py`.

Also run the full suite once to confirm no unrelated regressions:
```bash
python -m pytest backend/tests/ -v
```
