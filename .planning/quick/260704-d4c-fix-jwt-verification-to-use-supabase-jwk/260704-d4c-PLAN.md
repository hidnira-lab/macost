# Quick Task 260704-d4c: Fix JWT verification to use Supabase JWKS instead of hardcoded HS256 secret

## Context

While re-verifying the Phase 1 UAT fix (Supabase project provisioning, quick task 260704-bud), found that `backend/dependencies/auth.py` hardcoded HS256 + a manually-copied shared secret. The live Supabase project uses the newer "JWT Signing Keys" system (ES256 by default), which doesn't expose a copyable HS256 secret via the dashboard — the only usable HS256 key material would be internal to Supabase and never surfaced to us. Switching the signing key to a manually-created HS256 key also didn't help, since JWKS only ever publishes asymmetric public keys (confirmed by fetching `https://<project>.supabase.co/auth/v1/.well-known/jwks.json` directly — it never lists the HS256 key, only ES256).

## Task 1: Switch JWT verification to PyJWKClient

- File: `backend/dependencies/auth.py`
- Replace `os.environ["SUPABASE_JWT_SECRET"]` + `jwt.decode(token, secret, algorithms=["HS256"])` with `PyJWKClient(f"{SUPABASE_URL}/auth/v1/.well-known/jwks.json")` + `get_signing_key_from_jwt(token)`, decoding with `algorithms=["ES256", "RS256", "HS256", "EdDSA"]`.
- No longer depends on `SUPABASE_JWT_SECRET` env var at all — works regardless of which signing key type Supabase has active, as long as it's published via JWKS (any asymmetric type).
- User (Hidayat) to promote the project's original ES256 signing key back to "In Use" in the Supabase dashboard (it was temporarily switched to a manually-created HS256 key that JWKS never exposed).
