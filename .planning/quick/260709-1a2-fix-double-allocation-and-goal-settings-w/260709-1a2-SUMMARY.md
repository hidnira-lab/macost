# Quick Task 260709-1a2: Fix double-allocation and goal_settings weight-key validation Summary

**Two live-production backend correctness fixes: POST /api/allocations now rejects a second allocation on an already-allocated transaction, and PUT /api/goal-settings now enforces the exact 5-key SAW weights shape instead of only checking the sum.**

## Performance

- **Duration:** ~25 min
- **Tasks:** 2
- **Files modified:** 5

## Accomplishments

**Task 1 (Bug 1 — double allocation):**
- Added `_already_allocated()` helper to `backend/routers/allocations.py`, matching the file's existing `_transaction_not_found()`/`_goal_not_found()`/`_not_side_income()` pattern
- `confirm_allocation` now queries `alokasi` scoped by `transaksi_id` + `id_pengguna` after the two existing IDOR ownership checks and before the insert; raises 400 `VALIDATION_ERROR` if a row already exists
- Added `test_post_allocations_rejects_when_transaction_already_allocated`, mirroring the assertion style of the existing cross-user IDOR test (400 status, `VALIDATION_ERROR` code, `alokasi` table row count unchanged at 1)

**Task 2 (Bug 2 — weight-key validation):**
- Activated the previously-unused `GoalSettingsWeights` model in `backend/models/goal_settings.py` with `model_config = ConfigDict(extra="forbid")`
- Changed `GoalSettingsUpdate.weights` from `dict[str, float]` to `GoalSettingsWeights`
- Adapted the existing sum-tolerance `field_validator` (0.002 tolerance, unchanged logic) to operate on `v.model_dump().values()` since `v` is now a `GoalSettingsWeights` instance, not a raw dict
- `GoalSettingsResponse.weights` left as `dict[str, float]` — GET response shape untouched
- Updated `backend/routers/goal_settings.py`'s `.update(...)` call to pass `validated.weights.model_dump()` (plain dict) instead of the Pydantic model instance, since the Supabase client needs a JSON-serializable value for the jsonb column
- Added 3 tests to `backend/tests/test_goal_settings.py`: missing required key, typo'd key (`progress_gp`, the exact regression scenario from the bug report), and genuinely extra key — all assert 400 `VALIDATION_ERROR` and unchanged stored row

## Decisions Made
- Bug 1: no running-sum/partial-allocation cap was added on top of the duplicate-rejection check — confirmed while reading the router and `API_CONTRACT.md` §7 that no partial-allocation concept exists; rejecting any second POST for the same `transaksi_id` makes "sum of allocations <= nominal" trivially true (at most one allocation ever exists per transaction)
- Bug 2: fixed by activating the already-defined `GoalSettingsWeights` model rather than adding ad hoc key-set checks — smallest diff that closes the gap
- Sum-tolerance value (0.002) and its docstring rationale were preserved verbatim, only the input type it validates changed
