# Quick Task: Fix double-allocation and goal_settings weight-key validation

Two backend correctness bugs found during Phase 2 code review, both live in
production (`https://macost-production.up.railway.app`, real Supabase DB).
Both fixes are additive validation only — no response shape changes, no SAW
algorithm/weight changes, no changes to existing IDOR patterns.

## Context recap (from reading the code)

**Bug 1** — `backend/routers/allocations.py::confirm_allocation` (POST
`/api/allocations`, lines 92-157) inserts into `alokasi` unconditionally
after the two existing IDOR ownership checks (transaksi ownership lines
106-115, goal ownership lines 117-126). Nothing stops the same
`transaksi_id` from being allocated twice. `list_pending_allocations` (lines
204-257) already treats "no matching `alokasi.transaksi_id` row" as the
source of truth for pending/allocated status — so a duplicate `alokasi` row
not only creates phantom savings, it's invisible to that endpoint's own
logic (the transaction just silently stops appearing as pending, which is
the wrong reason for it to disappear).

The file's existing convention for an ownership check is:
```python
rows = (
    supabase.table("X")
    .select("*")
    .eq("id_column", value)
    .eq("id_pengguna", current_user_id)
    .execute()
    .data
)
if not rows:
    raise _some_not_found()
```
The new check follows the identical shape, querying `alokasi` scoped by
`transaksi_id` + `id_pengguna`, and raises 400 (not 404 — this isn't an
ownership failure, it's a state-conflict) if any row is found.

**Bug 2** — `backend/routers/goal_settings.py::update_goal_settings` (PUT
`/api/goal-settings`, lines 26-67) does `GoalSettingsUpdate(**body)`.
`GoalSettingsUpdate.weights` is typed `dict[str, float]` (`backend/models/goal_settings.py`
lines 17-19) — Pydantic accepts any string keys there, only the sum
`field_validator` (lines 21-41) runs. The 5-key `GoalSettingsWeights` model
(lines 6-14) exists but is never referenced. Because `dict[str, float]`
doesn't force the 5 real keys, a dict like
`{"personal_importance": 0.3, "progress_gp": 0.2, "saving_capacity": 0.2, "urgency": 0.15, "target_amount": 0.15}`
(typo'd `progress_gp`) sums to 1.0 and passes today, then later blows up
`saw_engine.rank_goals` (line 145: `v * multipliers[k]` iterates the
*weights* dict's own keys against `STRATEGY_MULTIPLIERS[strategy]`, so a
typo'd key throws `KeyError` there — actually on the `multipliers[k]`
lookup) or on `effective_weights[criterion]` (line 161) when a *real* key is
missing from the stored dict. Either way: 500 on every subsequent
`GET /api/goals` until the user PUTs valid weights again.

Fix: add `model_config = ConfigDict(extra="forbid")` to `GoalSettingsWeights`
(it already declares exactly the 5 real fields, so `extra="forbid"` rejects
both missing-and-substituted keys and genuinely extra keys), and change
`GoalSettingsUpdate.weights` type from `dict[str, float]` to
`GoalSettingsWeights`, adapting the sum validator to read `v.model_dump()`
(via `field_validator` on the nested model, or keep it on `GoalSettingsUpdate`
operating on `v.model_dump().values()`). Missing keys become a Pydantic
`ValidationError` (already caught in the router and turned into the
project's `400 VALIDATION_ERROR` shape) — same error code as the sum
mismatch, just a different underlying cause, which is fine per the router's
existing broad `except ValidationError` catch.

## Task 1 — Bug 1: reject double-allocation in POST /api/allocations

**File:** `backend/routers/allocations.py`

**Change:** In `confirm_allocation` (after the existing goal ownership
check at line 126, before building `insert_payload` at line 128), add a
new IDOR-style check querying `alokasi` for any existing row matching this
`transaksi_id` + `id_pengguna`:

```python
existing_alokasi = (
    supabase.table("alokasi")
    .select("id_alokasi")
    .eq("transaksi_id", body.transaksi_id)
    .eq("id_pengguna", current_user_id)
    .execute()
    .data
)
if existing_alokasi:
    raise HTTPException(
        status_code=status.HTTP_400_BAD_REQUEST,
        detail={
            "error": {
                "code": "VALIDATION_ERROR",
                "message": "Transaksi ini sudah dialokasikan sebelumnya",
            }
        },
    )
```

Add a small helper `_already_allocated()` mirroring the file's existing
`_transaction_not_found()`/`_goal_not_found()`/`_not_side_income()` helper
pattern (lines 14-37) instead of inlining the `HTTPException`, for
consistency with the rest of the file.

Do NOT touch the per-transaction amount-cap ("sum of allocations <= transaction's
nominal") part of the described bug — re-reading the current code, there is
exactly one `alokasi` row possible per `transaksi_id` once this check is in
place (any second POST is rejected outright), so the "sum exceeds nominal"
scenario collapses into "at most one allocation per transaction" and does
not need a separate running-sum check. This keeps the fix minimal per the
live-production constraint. If a genuinely partial-allocation flow exists
elsewhere (checked while reading the file — it does not: there is no partial
allocation concept anywhere in this router or in `API_CONTRACT.md` section
7), this scope decision would need revisiting, but it doesn't exist today.

**Do not change:** the response shape on success (still the exact dict at
lines 148-157), the two existing IDOR checks above it, or
`list_pending_allocations`/`skip_allocation`.

## Task 2 — Bug 2: enforce exact 5-key weights shape in PUT /api/goal-settings

**Files:** `backend/models/goal_settings.py`, `backend/routers/goal_settings.py`

**Change in `backend/models/goal_settings.py`:**

1. Add `ConfigDict` to the pydantic import (`from pydantic import BaseModel, ConfigDict, field_validator`).
2. Give `GoalSettingsWeights` `model_config = ConfigDict(extra="forbid")` —
   it already declares exactly the 5 real fields as required floats, so this
   makes it reject any dict with a missing real key or an extra/typo'd key
   (Pydantic v2 required-field-missing + extra-forbid both raise
   `ValidationError` on `GoalSettingsWeights(**incoming_dict)`).
3. Change `GoalSettingsUpdate.weights` from `dict[str, float]` to
   `GoalSettingsWeights`.
4. Move the existing sum-tolerance `field_validator` (lines 21-41,
   untouched logic — same 0.002 tolerance, same docstring) so it validates
   the nested `GoalSettingsWeights` model's field values. Since the
   validator is declared on `GoalSettingsUpdate.weights` and Pydantic v2
   runs field validators after the field's own type is constructed, the
   validator's incoming `v` will now be a `GoalSettingsWeights` instance,
   not a raw dict — update the body to `v.model_dump()` before summing:
   ```python
   @field_validator("weights")
   @classmethod
   def validate_weight_sum(cls, v: GoalSettingsWeights) -> GoalSettingsWeights:
       # ...same docstring...
       if abs(sum(v.model_dump().values()) - 1.0) >= 0.002:
           raise ValueError("weights must sum to 1.0")
       return v
   ```
5. `GoalSettingsResponse.weights` stays `dict[str, float]` (response shape,
   untouched — API_CONTRACT.md §6 GET response is unaffected).

**Change in `backend/routers/goal_settings.py`:**

`update_goal_settings` currently does
`supabase.table("goal_settings").update({"strategy": validated.strategy, "weights": validated.weights})`
(line 61). `validated.weights` is now a `GoalSettingsWeights` model
instance, not a plain dict — Supabase's client needs a JSON-serializable
dict for the `weights` jsonb column. Change that line to
`"weights": validated.weights.model_dump()`. Same for the return statement
at line 67 (`row["weights"]` — this reads back from the fake/real DB row,
which is already a plain dict post-round-trip, so line 67 itself needs no
change, only the `.update(...)` call's payload at line 61).

**Do not change:** the sum tolerance (0.002, still exact same check), the
`except ValidationError` catch-and-reraise-as-400 pattern (lines 41-52,
untouched — both the key-shape ValidationError and the sum ValueError
surface through the same catch, same 400 VALIDATION_ERROR shape), or
`GoalSettingsResponse`/`GET /api/goal-settings`.

**Tests to add** in `backend/tests/test_goal_settings.py` (same file,
following the existing `_build_app`/`_client_as`/`_patch_supabase`
conventions and the `DEFAULT_WEIGHTS` constant already defined at the top):

1. `test_put_goal_settings_missing_required_key_returns_400_validation_error`
   — seed a `goal_settings` row with `DEFAULT_WEIGHTS`. PUT with a `weights`
   dict that has only 4 of the 5 real keys (drop `target_amount`) but pad
   the remaining 4 so they still sum to ~1.0 (e.g.
   `{"personal_importance": 0.3, "progress_gap": 0.25, "saving_capacity": 0.25, "urgency": 0.2}`
   sums to 1.0). Assert `response.status_code == 400`,
   `response.json()["detail"]["error"]["code"] == "VALIDATION_ERROR"`, and
   the row in `fake_supabase_client` is unchanged (still `DEFAULT_WEIGHTS`)
   — mirrors the existing 0.95-sum test's unchanged-row assertion (lines
   109-139).
2. `test_put_goal_settings_typo_key_returns_400_validation_error` — same
   seed. PUT with a `weights` dict containing a typo'd key in place of a
   real one, e.g. `{"personal_importance": 0.225, "progress_gp": 0.219, "saving_capacity": 0.215, "urgency": 0.178, "target_amount": 0.163}`
   (`progress_gp` typo, still 5 keys, still sums to ~1.0). Assert
   `response.status_code == 400`,
   `response.json()["detail"]["error"]["code"] == "VALIDATION_ERROR"`, row
   unchanged. This is the exact regression scenario from the bug report —
   the one that used to silently pass and corrupt stored weights.
3. `test_put_goal_settings_extra_key_returns_400_validation_error` — same
   seed. PUT with all 5 real keys plus one extra (e.g.
   `"extra_criterion": 0.0`), still summing to ~1.0 across the real 5 (the
   extra key's value can be non-zero as long as the real 5 alone don't need
   to sum to 1.0 — but simplest is extra key value 0.0 so the real 5 stay
   at the known-good `DEFAULT_WEIGHTS`). Assert 400 VALIDATION_ERROR, row
   unchanged.

Also re-run the two pre-existing PUT tests (`..._0_95_returns_400...` and
`..._within_tolerance_succeeds...`) to confirm they still pass unmodified —
the 0.95-sum test uses exactly the 5 real keys already so `extra="forbid"`
doesn't affect it, and the exactly-1.0 test uses `DEFAULT_WEIGHTS` verbatim.
