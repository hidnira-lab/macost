# Quick Task: Backport amount-validation guard into AllocationSuggestionModal

Found during Phase 2 code review, stopgap approved by Day (full consolidation
of `SmartAllocationModal.tsx` vs `AllocationSuggestionModal.tsx` deferred
post-demo).

## Context (re-verified before planning, not just from the review report)

`apps/web/components/AllocationSuggestionModal.tsx` (used by
`apps/web/app/transactions/page.tsx`, the desktop/list transaction flow) has
**no user-editable amount** — `handleConfirm()` always sends
`suggestion.suggested_amount` verbatim, never a custom value. This means the
`allocationAmount > incomeAmount` guard that exists in
`SmartAllocationModal.tsx` (lines 152-159, guarding a `customAmount` the user
can type into an input) **does not have a direct equivalent need here** —
there is no user input path that could produce an amount exceeding income,
since `suggested_amount` is always server-computed as 35% of the transaction
(`SUGGESTED_PCT = 35` in `backend/services/allocation_service.py`), which by
construction is always ≤ income.

The actual, real gap in `AllocationSuggestionModal.tsx` is narrower than the
review's headline suggested:
- Line 40: `if (!suggestion.suggested_goal_id || !suggestion.suggested_amount) return`
  already guards against a missing/zero amount — but it does so **silently**
  (a bare `return`, no error shown, no visual feedback). If this guard ever
  trips (malformed API response, `suggested_amount: 0`), the user clicks
  "Konfirmasi Alokasi" and nothing visibly happens — confusing, not unsafe.
- The "Konfirmasi Alokasi" button itself is never `disabled` based on
  suggestion validity — only on `busy` (in-flight request state). A user
  could click it while the same silent-no-op condition applies with no
  visual indication the button won't do anything.

## Task 1: Make the existing guard visible + disable the button when suggestion is invalid

**File:** `apps/web/components/AllocationSuggestionModal.tsx`

**Change:**
1. In `handleConfirm()`, replace the silent `if (!suggestion.suggested_goal_id || !suggestion.suggested_amount) return`
   with a version that sets a visible error via the existing `error` state
   (same `setError(...)` pattern already used for the API-failure catch
   block below it), e.g.:
   ```ts
   if (!suggestion.suggested_goal_id || !suggestion.suggested_amount || suggestion.suggested_amount <= 0) {
     setError('Saran alokasi tidak valid. Coba tutup dan buka kembali.')
     return
   }
   ```
2. Compute a derived boolean near the top of the component (after the
   `if (!open) return null` guard), e.g.
   `const isSuggestionValid = Boolean(suggestion.suggested_goal_id) && (suggestion.suggested_amount ?? 0) > 0`,
   and use it to extend the "Konfirmasi Alokasi" button's existing
   `disabled={busy}` to `disabled={busy || !isSuggestionValid}` (only inside
   the `suggestion.has_active_goal` branch — the `!has_active_goal` branch's
   buttons are unrelated and must not change).
3. Do NOT add a `customAmount`/editable-amount feature to this component —
   that's the larger consolidation decision Day explicitly deferred, not
   part of this stopgap.
4. Do NOT change `SmartAllocationModal.tsx` — it already has the guards;
   this task only touches `AllocationSuggestionModal.tsx`.
5. Do NOT change the request payload, endpoint calls, or response handling
   — only add the validity check + visible error + button disabled state.

**Verify:**
```
cd apps/web && npx tsc --noEmit
cd apps/web && npm run lint
```
Both must show no NEW errors/warnings versus the current baseline (a
pre-existing `react-hooks/set-state-in-effect` error in
`SmartAllocationModal.tsx` is known-unrelated — confirm it's still the only
pre-existing issue, don't try to fix it as part of this task).

## Constraints

- Single small file, single small change — visible error + button disabled
  state only. No new props, no new endpoints, no editable-amount feature.
- This is explicitly a stopgap per Day's decision, not the full
  consolidation the code review recommended — do not merge the two
  components or delete either one.
