# Quick Task Summary: Backport amount-validation guard into AllocationSuggestionModal

## What changed

`apps/web/components/AllocationSuggestionModal.tsx` — single file, two small
edits, exactly per plan:

1. **Visible error instead of silent no-op.** `handleConfirm()`'s guard
   (`if (!suggestion.suggested_goal_id || !suggestion.suggested_amount) return`)
   now also treats `suggested_amount <= 0` as invalid, and sets a visible
   error via the existing `error` state (`setError('Saran alokasi tidak
   valid. Coba tutup dan buka kembali.')`) instead of returning silently.
2. **Button disabled on invalid suggestion.** Added a derived
   `isSuggestionValid` boolean (`Boolean(suggestion.suggested_goal_id) &&
   (suggestion.suggested_amount ?? 0) > 0`), and extended the "Konfirmasi
   Alokasi" button's `disabled={busy}` to `disabled={busy ||
   !isSuggestionValid}`. Only the `has_active_goal` branch was touched — the
   "Belum ada goal aktif" branch's buttons are unchanged.

No changes to `SmartAllocationModal.tsx`, no new props, no editable-amount
feature, no request/response/endpoint changes — scope matched the plan
exactly.

## Why (context)

Found during the Phase 2 code review: `AllocationSuggestionModal.tsx` (used
by the desktop/list transaction flow) had no user-editable amount, so the
`allocationAmount > incomeAmount` guard from `SmartAllocationModal.tsx` had
no direct equivalent need (server-computed `suggested_amount` is always ≤
income by construction). The real gap was narrower: an existing silent
guard plus a Confirm button that was never disabled for an invalid
suggestion. Day approved this narrow stopgap; the larger
component-consolidation the review recommended is explicitly deferred
post-demo.
