# Quick Task: Remove tipe_transaksi from frontend transaction payloads

Found during Phase 2 code review. CLAUDE.md rule 2 / `.claude/CLAUDE.md` says
frontend must never determine source labeling — only send `kategori_id`,
backend derives `tipe_transaksi`/`source_label` server-side. Confirmed
backend already fully ignores any `tipe_transaksi` sent by the client
(`backend/routers/transactions.py` create + update paths both use a
locally-derived value, never `body.tipe_transaksi` — verified true, no
backend change needed). But the frontend still sends it in 3 places, one of
which sends an actively wrong hardcoded value.

## Context

1. `apps/web/lib/api/types.ts:106,124` — `tipe_transaksi: string` is a
   **required** field on both `TransactionCreateRequest` and
   `TransactionUpdateRequest`. This is backwards from the documented
   contract (frontend only needs `kategori_id`) and forces every caller to
   invent a value. Note `types.ts` already has a doc-comment a few lines up
   warning against doing exactly this for the `source` field — the same
   rule was just never applied to `tipe_transaksi`.
2. `apps/web/components/TransactionForm.tsx:103` — sends
   `tipe_transaksi: selectedCategory?.tipe ?? 'Pengeluaran'`. This value
   happens to be correct today (mirrors the category's real type), but it's
   dead weight now enforced by the required type above.
3. `apps/web/app/transactions/new/page.tsx:114` — sends a **hardcoded,
   lowercase, always-'pengeluaran'** value regardless of whether the
   transaction is actually income or expense. This is live wrong data
   (doesn't even match the backend's real enum casing `"Pemasukan"`/
   `"Pengeluaran"`), currently harmless only because the backend ignores it.

## Task 1: Remove tipe_transaksi from the frontend request contract and both call sites

**Files:**
- `apps/web/lib/api/types.ts`
- `apps/web/components/TransactionForm.tsx`
- `apps/web/app/transactions/new/page.tsx`

**Change:**
1. In `apps/web/lib/api/types.ts`, remove the `tipe_transaksi: string;` line
   from both `TransactionCreateRequest` (line 106) and
   `TransactionUpdateRequest` (line 124). Do not touch any other field on
   either interface. Do not touch `TransactionResponse`/any response type —
   `tipe_transaksi` and `source_label` are legitimate **response** fields
   (server-computed, per API_CONTRACT.md §4) and must stay there unchanged.
2. In `apps/web/components/TransactionForm.tsx`, remove the
   `tipe_transaksi: selectedCategory?.tipe ?? 'Pengeluaran',` line (103) from
   the payload object being built. Leave the comment above it (line 95,
   explaining why it's not needed) if still accurate, or tighten it to
   reflect that the field is no longer sent at all (not just "auto-derived
   server-side" but "not sent by the frontend at all").
3. In `apps/web/app/transactions/new/page.tsx`, remove the
   `tipe_transaksi: 'pengeluaran',` line (114) from the payload object.
4. After edits, run `grep -rn "tipe_transaksi" apps/web/` and confirm the
   only remaining matches are: (a) response-type definitions in
   `apps/web/lib/api/types.ts` (the response interfaces, untouched by this
   task), and (b) any place that *reads* `tipe_transaksi` off an API
   *response* to render UI (e.g. showing "Pemasukan"/"Pengeluaran" labels in
   a transaction list) — those are legitimate reads of server data and must
   NOT be touched.

**Do not change:** any backend file (already confirmed correct — this is a
frontend-only cleanup), any other field on the two request interfaces, any
response-type interface, any UI copy/labels that read `tipe_transaksi` from
an already-fetched transaction object.

**Verify:**
```
cd apps/web && npx tsc --noEmit
```
Must complete with zero new type errors (removing a required field that
callers were already satisfying can't introduce errors; the goal is
confirming no other file was silently relying on
`TransactionCreateRequest.tipe_transaksi`/`TransactionUpdateRequest.tipe_transaksi`
being present — if `tsc` reports an error in a file this task didn't touch,
that file has a real reliance on the now-removed field and needs its own
fix, report it rather than silently patching around it).

Also run existing frontend lint if configured:
```
cd apps/web && npm run lint
```
Must pass (or show only pre-existing warnings unrelated to this change).

## Constraints

- No backend changes — verified backend already ignores this field.
- No changes to any *response* type or any UI code that reads
  `tipe_transaksi`/`source_label` from a fetched transaction — only the
  three *outgoing request* sites listed above.
- Minimal diff — 2 line removals in `types.ts`, 1 line removal each in the
  two component files, plus the one comment tightening.
