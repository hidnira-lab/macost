<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> `POST /api/transactions` | User-controlled `kategori_id`/`nominal`/`dompet_id` cross into a write that decides money-labeling logic |
| Browser -> `GET /api/transactions` | User-controlled filter query params cross into a per-user-scoped read |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-02 | Tampering | `POST /api/transactions` (`tipe_transaksi`/`source_label` override attempt) | high | mitigate | Handler derives both fields exclusively from the looked-up `kategori` row; `TransactionCreate.tipe_transaksi` is accepted but structurally never read when building the insert payload (Pattern 1 / Pitfall 1) |
| T-2-01 | Tampering / Information Disclosure | `GET /api/transactions` (IDOR via `category_id`/pagination params) | high | mitigate | Every query unconditionally includes `.eq("id_pengguna", current_user_id)` in addition to any caller-supplied filter, mirroring `wallets.py`'s double-scoping pattern |
| T-2-04 | Tampering -> Denial of Service | `nominal <= 0` | medium | mitigate | Pydantic `Field(gt=0)` rejects at the API boundary with `422` before any DB write or downstream SAW computation is reached |

</threat_model>
