<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> `PUT /api/goal-settings` | User-controlled weights cross into a validated write that ultimately biases every goal ranking |
| Browser -> Goals CRUD endpoints | Standard per-user IDOR boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-2-03 | Tampering | `PUT /api/goal-settings` weight-sum manipulation | high | mitigate | Pydantic `field_validator` rejects any weight set outside `abs(sum-1.0) < 0.001`, returned as structured `400 VALIDATION_ERROR` |
| T-2-01 | Tampering / Information Disclosure | Goals/allocation_history IDOR | high | mitigate | Every goal/alokasi query double-scopes `.eq(id, ...).eq("id_pengguna", current_user_id)` |
| T-2-02 | Tampering | Client attempting to send `rank`/`progress_pct`/`nominal_terkumpul`/`skor_kepentingan` on `POST`/`PUT /api/goals` | high | mitigate | `GoalCreate`/`GoalUpdate` Pydantic models structurally exclude these fields entirely — they only ever appear in `GoalResponse`, computed server-side by `fetch_and_rank_goals` |
| T-2-04 | Tampering -> Denial of Service | `nominal_target <= 0`, deadline in the past | medium | mitigate | Pydantic `Field(gt=0)` + explicit deadline-future validator reject at the API boundary before reaching `goal_service`/`saw_engine` |

</threat_model>
