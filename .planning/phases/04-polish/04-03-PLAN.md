<objective>
Deliver OFF-01 (offline write queue + auto-sync) and OFF-02 (sync status indicator) — the frontend half of Phase 4's offline reliability work, built on top of 04-01's backend idempotency guarantee.

Purpose: Per D-01 (04-CONTEXT.md), offline scope is broadened beyond plain transactions to include goal create/edit and allocation confirm/skip, all queued in IndexedDB and synced automatically the moment the browser/WebView reports it's back online (D-04) — with the non-negotiable rule that a side-income transaction's Smart Allocation suggestion is deferred until after sync completes (D-02), never shown or auto-executed at enqueue time.
Output: An `idb`-backed offline queue module (`lib/offline/`), a global `SyncStatusIndicator` component, and the 3 write-path call sites (transaction create, goal create/edit, allocation confirm/skip) updated to fall back to the queue when offline or when the network call fails.

Depends on 04-01 because `queue.ts`'s `replayItem()` sends `idempotency_key` in each request body — a field that only exists on the backend's request models after 04-01 lands.

Contingency note (D-10): if time runs short before 2026-07-14, the documented first cut is narrowing this plan's scope back to transactions-only (dropping the goal/allocation queueing added below in Task 3, which restores the literal OFF-01 requirement text) — pixel art (04-02) and the 5 fixed states are NOT the fallback target; this plan implements the full D-01-broadened scope as the primary goal, with D-10's narrowing as an explicit, pre-approved descope path only if needed.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| IndexedDB queue -> network sync | Queued payloads are user's own prior input, replayed later — same trust level as if submitted live; idempotency_key (client UUID) crosses this boundary and must be validated server-side (handled in 04-01) |
| navigator.onLine signal -> sync trigger | Untrusted/unreliable signal (Pitfall 2) — must never be treated as a guarantee of actual connectivity, only a trigger to attempt sync |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-4-07 | Repudiation | Sequential sync replay ordering | medium | mitigate | sync() always awaits each replayItem() sequentially in a for...of loop, never Promise.all — prevents an allocation-confirm racing ahead of its referenced goal's own queued creation (Pitfall 3) |
| T-4-08 | Tampering | Deferred allocation-suggestion modal (D-02 / suggest-and-confirm rule) | critical | mitigate | The suggestion fetch+modal only fires inside sync()'s per-item success handler for kind:'transaction', never at enqueue time and never auto-confirmed — preserves CLAUDE.md rule #4 (never auto-execute) even across the offline path |
| T-4-09 | Information Disclosure | IndexedDB queue data at rest (unencrypted local storage) | low | accept | Consistent with the existing app's risk posture — JWT already stored unencrypted in localStorage/tauri-plugin-store; no new requirement for offline-queue encryption per CONTEXT.md/ROADMAP.md |
| T-4-10 | Denial of Service | Malformed/stuck queue item blocking sync forever | medium | mitigate | A failing item's attempts counter increments and the item stays queued, but the for...of loop always continues to the next item on catch — one bad item never blocks the rest of the queue from syncing |

</threat_model>
