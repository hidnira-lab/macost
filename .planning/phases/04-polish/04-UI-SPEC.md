# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

This phase has two net-new UI surfaces with **no corresponding Figma frame** (per CLAUDE.md, Figma is normally the source of visual truth for the app's 21-page design system, but VIS-01's pixel art and OFF-02's sync indicator are explicitly out of scope of the existing Figma file per `04-CONTEXT.md`). This contract instead derives visual consistency from the **already-implemented Palette A design language** found in `apps/web/app/globals.css`, `apps/web/components/BottomNav.tsx`, `apps/web/components/SuggestionCard.tsx`, and `apps/web/app/goals/page.tsx` / `apps/web/app/goals/[id]/page.tsx` — all shipped in Phase 2/3. Pixel art asset specifications (count, dimensions, per-state motif, file naming) are already fully locked in `04-CONTEXT.md` `<specifics>` and are **not re-specified here** — this contract covers only placement, layout integration, and rendering treatment for both surfaces.

Offline queue mechanics (OFF-01) themselves have no UI surface beyond the sync indicator (OFF-02) — no separate spec section needed for the queue internals.

## Spacing Scale

Declared values (must be multiples of 4) — inherited from the app's existing (undeclared but consistently-used) spacing rhythm, observed directly in `goals/page.tsx` / `SuggestionCard.tsx`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (e.g. `gap-1` in footer action buttons) |
| sm | 8px | Compact spacing inside cards (`gap-2`, `mt-1`) |
| md | 16px | Default card padding (`px-4 py-4`), section gaps (`mb-4`) |
| lg | 24px | Section-to-section spacing (`mb-6`), card padding on larger elements (`px-5`) |
| xl | 32px | Page-level top padding (`pt-8`) |

Exceptions:
- Pixel art sprite sizes (48px card thumbnail, 128px detail hero) are **not** spacing-scale values — they are fixed asset display dimensions per `04-CONTEXT.md` Asset Guide, exempt from the 8-point spacing rule (same exception class as the existing `44px` icon-only touch target convention).
- Sync status indicator badge uses `8px` (sm) internal padding and `4px` (xs) icon-text gap — both on-scale, no exception needed.

## Color

Palette A (already locked app-wide per STATE.md quick-task `260706-jaq` round 3 — the single source of truth for all pages, superseding an earlier dark-theme draft):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fcfcfc` | Page background (`bg-[#fcfcfc]` / inline `backgroundColor: '#fcfcfc'`) |
| Secondary (30%) | `#ffffff` + `rgba(30,30,30,0.15)` border | Cards, nav bar, panels |
| Accent (10%) | `#298dff` (blue) / `#ff8929` (orange) | See "Accent reserved for" below |
| Destructive | `#ba1a1a` (text/icon) on `rgba(255,218,214,0.5)` (surface) | Destructive actions only (e.g. existing Delete goal button) |

Accent reserved for:
- `#298dff` (blue) — primary interactive elements: active nav tab background (`BottomNav.tsx`), primary text links/buttons ("Tinjau Saran", strategy link), page-title text on goal detail.
- `#ff8929` (orange) — progress/priority signaling: progress bar fill, priority-#1 badge, progress percentage text.
- **New for this phase — Sync status indicator color contract** (not a new accent color, reuses existing semantic tokens already in the codebase):
  - `synced` state → `#059669` (existing "On Track" green, `rgba(16,185,129,0.12)` background) — already used in `goals/page.tsx` status pill.
  - `syncing` state → `#298dff` (existing blue accent) — reuses accent, does not introduce a new hue.
  - `offline` state → `rgba(30,30,30,0.65)` text on `rgba(30,30,30,0.05)` background (existing neutral/muted token, same as the `#{goal.rank}` badge in `goals/page.tsx`) — deliberately neutral/muted, not alarming, since offline is an expected non-error state per D-04's fully-automatic sync design.
  - No new red/error color introduced for `offline` — offline is not treated as a failure state in this app's UX (per RESEARCH.md Pitfall 2, a failed sync is silently retried, not surfaced as an error).

## Component Spec: GoalPixelArt (VIS-01)

**Component name:** `GoalPixelArt` (illustrative name from RESEARCH.md — exact naming is executor's call, but keep in `apps/web/components/`).

**Asset source:** 5 pre-drawn PNGs at `apps/web/public/pixel-art/goal-plant-{0,25,50,75,100}.png` (already fully specified in `04-CONTEXT.md` — dimensions 64×64px source, transparent background — not re-specified here).

### Placement — Goal Detail Page (`apps/web/app/goals/[id]/page.tsx`)

Current layout (read from file): Header → error banner → **Hero Graphic** (256px gradient block with a generic `Target` icon, orange gradient, currently placeholder-style) → Progress Card → Allocation History → Footer actions.

**Contract:** Replace the existing placeholder Hero Graphic's centered icon with the pixel art sprite, keeping the same gradient card shell (do not remove the orange gradient background — it remains as a decorative backdrop).

- **Placement:** Inside the existing Hero Graphic container (`height: 256px`, orange gradient background), centered both axes — same position currently occupied by the `<Target className="w-20 h-20 text-white/30" />` icon.
- **Size:** 128px × 128px (per `04-CONTEXT.md` "detail" size).
- **Spacing:** No additional margin needed — center via existing `flex items-center justify-center` on the parent container (already present).
- **Coexistence with progress_pct:** The numeric `{goal.progress_pct}%` value and progress bar remain in the **Progress Card** below (unchanged, not duplicated onto the hero). The pixel art is purely the hero visual; the existing progress bar/percentage stays the single source of numeric truth. Do not add a second percentage label near the pixel art.
- **CSS treatment:** `image-rendering: pixelated` inline style (plus `-moz-crisp-edges` fallback) directly on the `<img>` element, per RESEARCH.md Pitfall 5.
- **Element choice:** Plain `<img>` (not `next/image`) — consistent with `images.unoptimized: true` already set repo-wide, avoids any risk of Next's Image wrapper interfering with pixel-perfect scaling.
- **Alt text:** Use the Copywriting Contract's alt-text formula above, populated with the current goal's resolved state and motif label.

### Placement — Goal List / Card View (`apps/web/app/goals/page.tsx`)

Current layout (read from file): two card variants exist — **Priority #1 Card** (larger, orange corner blob, `Sparkles` badge) and **Standard Goal Card** (compact, rank badge `#{goal.rank}`). Both currently show a circular icon badge (`iconRingClass`, 40px/`w-10 h-10`) picked from a rotating icon+color palette (`goalVisual(index)`), positioned top-left of the card content, next to the goal name.

**Contract:** Add the pixel art thumbnail **alongside**, not replacing, the existing icon badge — the icon badge is a category-agnostic visual identifier already established across the list; removing it would be a bigger visual change than this phase's scope warrants. Mount the pixel art thumbnail as an additional small element.

- **Placement — Priority #1 Card:** Add the 48px pixel art thumbnail in the top-right area, near the existing `Sparkles`/"Prioritas #1" badge — specifically, stack it directly **above** the "Prioritas #1" badge, right-aligned, so the top row becomes: `[icon badge + name/amounts]` ... `[pixel art thumbnail (48px), then badge below it]`. Use a `flex flex-col items-end gap-1` wrapper for the two right-aligned elements.
- **Placement — Standard Goal Card:** Add the 48px pixel art thumbnail to the **right** of the existing rank badge (`#{goal.rank}`), same row, both right-aligned and stacked vertically (`flex flex-col items-end gap-1`): pixel art thumbnail on top, rank badge below.
- **Size:** 48px × 48px (per `04-CONTEXT.md` "card" size).
- **Spacing:** `4px` (xs token) gap between the pixel art thumbnail and the badge/rank element stacked below it, using existing Tailwind `gap-1` utility (already used elsewhere in this file, e.g. `gap-1` in `SuggestionCard.tsx`'s button).
- **Coexistence with progress_pct:** Each card already shows a full-width progress bar + percentage below the header row — unchanged. The pixel art thumbnail is an additional at-a-glance visual, not a replacement for the numeric/bar progress display.
- **CSS treatment:** Same `image-rendering: pixelated` + plain `<img>` as the detail page.
- **Alt text:** Same formula, scoped to that card's goal.

### Rendering Logic (shared, both placements)

```
STATES = [0, 25, 50, 75, 100]
state = highest value in STATES where progress_pct >= state (clamped 0-100)
src = `/pixel-art/goal-plant-${state}.png`
```
(Matches RESEARCH.md `pixelArtState()` reference implementation — executor may reuse verbatim.)

## Component Spec: Sync Status Indicator (OFF-02)

**Component name:** `SyncStatusIndicator` (suggested name — executor's call).

### Placement Decision

**Mount near `BottomNav.tsx`, rendered as a slim bar directly above the nav bar** (not inside `BottomNav.tsx` itself — keep it a separate, independently-mountable component so it can be included from the root layout or per-page without coupling to nav internals). Rationale: `BottomNav` is the only persistent, always-visible chrome across all authenticated pages (confirmed via `activeTab` prop usage in `goals/page.tsx`), making it the natural anchor point, consistent with RESEARCH.md's own recommendation. A header-mounted badge was considered and rejected — `goals/page.tsx`'s sticky header already has 3 elements (logo, title, notification bell) at a 64px height; adding a 4th would crowd it. A toast/banner was rejected — OFF-02 needs to be a persistent ambient indicator, not a transient notification (offline/syncing states can last indefinitely).

### Visibility Rule

- **Only rendered when state is NOT `synced`.** In the default (synced, online, empty queue) case, the indicator is **absent entirely** — no persistent "Tersinkron" chip cluttering the UI at all times. This matches the "favor simple, low-effort" guidance in phase notes and avoids permanent visual noise for the common-case (online, synced) user.
- Appears the moment state becomes `offline` (browser/WebView reports disconnected, or a sync attempt fails) or `syncing` (queue drain in progress).
- On transition to `synced` (queue empty, online confirmed): show the "Tersinkron" state **briefly** (2 seconds) as a success confirmation, then auto-hide. This gives the user closure that their offline data landed, without becoming permanent chrome.

### Visual Treatment

A slim full-width bar, not a floating badge/pill — sits flush above `BottomNav`, same visual weight class as a system status strip:

- **Layout:** `fixed bottom-[56px] left-0 right-0 z-40` (positioned directly above BottomNav's own `fixed bottom-0` — adjust the exact offset to BottomNav's actual rendered height at build time), full width, centered content, `max-w-2xl mx-auto` wrapper matching the app's existing content-width convention (seen in `BottomNav.tsx`'s own `max-w-2xl mx-auto`).
- **Height/padding:** `py-2 px-4` (8px vertical, 16px horizontal) — compact, single-line.
- **Per-state colors (reusing existing semantic tokens, see Color section):**

| State | Background | Text/Icon Color | Icon (lucide-react) |
|-------|------------|------------------|----------------------|
| `offline` | `rgba(30,30,30,0.05)` | `rgba(30,30,30,0.65)` | `WifiOff` |
| `syncing` | `rgba(41,141,255,0.1)` | `#298dff` | `RefreshCw` (with `animate-spin` Tailwind class) |
| `synced` (transient, 2s) | `rgba(16,185,129,0.12)` | `#059669` | `Check` |

- **Typography:** Label role (12px `text-xs`, Helvetica, `font-semibold`) — matches existing status-pill text treatment (e.g. "On Track"/"Active" pills in `goals/page.tsx`).
- **Border:** `1px solid` matching the icon/text color at low opacity, OR no border (flat fill) — keep flat fill only, consistent with `SuggestionCard.tsx`'s icon-badge treatment (no border on colored badges in this codebase, only on white card surfaces).
- **Shape:** Full-width bar has **no border-radius** (edge-to-edge, sits flush against BottomNav above it) — this deliberately differs from the app's usual `rounded-xl`/`rounded-full` card language, signaling "system status strip" rather than "content card."

### Transition / Animation

Per phase notes ("favor simple, low-effort... plain CSS transitions rather than complex animation"):

- State-to-state color/background change: plain CSS `transition-colors duration-200` (Tailwind utility) — no custom keyframe animation.
- `syncing` icon: `animate-spin` (existing Tailwind utility, already available, zero new CSS).
- Mount/unmount (appear on offline/syncing, disappear after synced-confirmation window): simple opacity fade via `transition-opacity duration-200` — no slide/bounce/complex easing.
- No skeleton loaders, no multi-step animation sequences.

### Copy (Bahasa Indonesia — see Copywriting Contract above for full detail)

- Offline: "Offline"
- Syncing: "Menyinkronkan..." (+ optional "{n} item menunggu" secondary line if queue length > 0)
- Synced (transient): "Tersinkron"
