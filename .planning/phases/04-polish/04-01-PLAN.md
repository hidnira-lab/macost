<objective>
Build the backend idempotency foundation that makes offline-queued writes (transactions, goals, allocations) safe to retry without creating duplicate financial records — the prerequisite for the frontend offline queue (04-03) to sync safely.

Purpose: OFF-01's D-03 decision requires every offline-queued write to carry a client-generated UUID idempotency key; the backend must recognize a retried key and return the original result instead of inserting again. This plan delivers that server-side guarantee in isolation, fully testable without any frontend code. Scheduled first (Wave 1) per D-09's work-order decision — offline queue + idempotency + conflict handling is the riskier/more technical piece of Phase 4, tackled while there's still schedule slack; pixel art (04-02) is comparatively mechanical and safe to finish closer to the wire.
Output: A live Postgres migration (idempotency_key column + partial UNIQUE index on transaksi/goal/alokasi), a shared `check_idempotency()` helper, idempotency-aware POST/PUT handlers in the 3 affected routers, and pytest coverage for the create-once / retry-noop / cross-user-isolation behaviors.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client (offline queue, Phase 04-03) -> API | Client-generated UUID (idempotency_key) crosses this boundary untrusted — must be validated as a well-formed string and always scoped to the authenticated user server-side |
| API -> Database | check_idempotency() lookups and inserts both cross this boundary; the partial UNIQUE index is the actual atomic safety net, the SELECT-then-INSERT check is only an optimization |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-4-01 | Tampering | POST /api/transactions, /api/goals, /api/allocations idempotency check | high | mitigate | check_idempotency() always double-scopes by id_pengguna AND idempotency_key (never a bare key lookup) — closes cross-user IDOR via a guessed/leaked UUID; enforced identically across all 3 routers via the shared helper |
| T-4-02 | Repudiation | Retried sync of a queued write (network flake, app restart mid-sync) | high | mitigate | Postgres partial UNIQUE index (id_pengguna, idempotency_key) is the atomic duplicate-prevention mechanism; the application-level SELECT check is only a fast-path optimization, not the sole guard |
| T-4-03 | Tampering | Migration 008 nullability | medium | mitigate | Column stays nullable + partial index (WHERE idempotency_key IS NOT NULL) so existing rows and every non-offline write path are unaffected — verified by all pre-existing tests in the 3 affected test files continuing to pass unmodified |
| T-4-04 | Denial of Service | Malformed idempotency_key value reaching the DB query | low | accept | Pydantic field is a plain `str | None` (not strict UUID4) for MVP timeline reasons; a malformed value simply never matches an existing row (falls through to normal insert) — no crash path, worst case is a wasted lookup |

</threat_model>
