<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser PDF upload -> FastAPI multipart endpoint | Untrusted user-supplied binary content; client-declared MIME type is not trustworthy |
| FastAPI -> Gemini API | Server-to-server call forwarding user-supplied PDF bytes to a third-party LLM |
| Duplicate-detection query -> transaksi table | Must never leak or match against another user's transaction rows |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-3-14 | Information Disclosure | Duplicate-detection query missing id_pengguna scope, leaking cross-user match signals | high | mitigate | `flag_duplicates()` scopes its Supabase query with `.eq("id_pengguna", user_id)`, verified by a dedicated IDOR-safety test case (Task 1) |
| T-3-15 | Denial of Service | Oversized PDF upload wasting memory/Gemini quota | high | mitigate | Explicit 50MB size cap and `%PDF` magic-number validation BEFORE the Gemini call |
| T-3-16 | Tampering | Batch import silently trusting a client-supplied tipe_transaksi/source_label per row | high | mitigate | `import-batch` re-derives tipe_transaksi/source_label per-row from the looked-up kategori, identical to `create_transaction`'s rule — never reads it from the request body |
| T-3-17 | Elevation of Privilege | Extracted/flagged rows imported without explicit per-row user confirmation | high | mitigate | Structural: `upload-statement` never writes to `transaksi`; only `import-batch`, reached exclusively via the review table's explicit "Import Terpilih" action with only checked rows, can write |

</threat_model>
