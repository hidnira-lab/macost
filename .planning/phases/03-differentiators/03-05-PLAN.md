<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser file upload -> FastAPI multipart endpoint | Untrusted user-supplied binary content crosses this boundary; client-declared MIME type is not trustworthy |
| FastAPI -> Gemini API | Server-to-server call forwarding user-supplied image bytes to a third-party LLM |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-3-10 | Denial of Service | Oversized or malformed file upload wasting memory/Gemini quota | high | mitigate | Explicit size cap (10MB) and magic-number validation BEFORE the Gemini call, verified by dedicated test cases asserting `extract_receipt` is never invoked on a rejected upload |
| T-3-11 | Tampering | Client-supplied `Content-Type` header spoofed to bypass file-type gating | medium | mitigate | Magic-number byte-sniffing on the actual file content, never trusting `UploadFile.content_type` alone, per RESEARCH.md Pitfall 3 |
| T-3-12 | Elevation of Privilege / Tampering | Extracted data auto-saved without user review, bypassing suggest-and-confirm | high | mitigate | Structural: scan-receipt endpoint has zero write path to the `transaksi` table; the only save path is the existing, already-reviewed `POST /api/transactions`, reached exclusively through the review form's explicit submit |
| T-3-13 | Repudiation / Availability | Automatic retry on Gemini failure amplifying cost/latency or masking a persistent failure | medium | mitigate | D-01 explicitly forbids auto-retry; Task 1's fallback path returns a terminal `extracted: false` response and Task 2's frontend routes to manual input rather than re-invoking the scan endpoint automatically |

</threat_model>
