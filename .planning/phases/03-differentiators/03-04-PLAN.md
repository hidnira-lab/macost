<objective>
Deliver SAW-04 (manual weight adjustment with live re-rank preview, ±0.002 tolerance validation) and SAW-05 (reset to n=62 research defaults) as one vertical slice: a new backend preview endpoint that reuses the already-tested `saw_engine.rank_goals()` and `GoalSettingsUpdate` validator, plus a frontend weight-editor page.

Per D-05 (locked): manual numeric entry for the five criteria with a live re-rank preview and a "Reset ke default" action, using the ±0.002 sum tolerance (not ±0.001) because the locked n=62 defaults sum to 0.999. Per RESEARCH.md Pattern 3 and the "Don't Hand-Roll" table, the live re-rank preview must call the backend with candidate weights rather than porting SAW scoring math to TypeScript — two independent scoring implementations would drift (rounding, tie-break order, strategy multipliers). This plan is fully independent of the Gemini integration work in 03-01/03-05/03-06/03-07 — `backend/models/goal_settings.py`'s ±0.002 validator already exists from Phase 2 (`SAW-04 | ±0.002 tolerance already covered | existing`, per 03-RESEARCH.md's Phase Requirements table); this plan only adds the preview endpoint, the reset default (already available as `DEFAULT_WEIGHTS` in `goal_settings_service.py`), and the frontend UI.

Purpose: Directly targets the PRD's SAW-04/05 requirement, letting a power user tune their own priority weighting instead of being locked to the survey defaults, while the live preview keeps the UX honest about the ranking impact before committing.
Output: `POST /api/goal-settings/preview` endpoint, `SawWeightEditor.tsx` component, `apps/web/app/goal-settings/page.tsx` page.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Authenticated user -> POST /api/goal-settings/preview | New write-shaped endpoint (POST) that does NOT persist — must not be confused with PUT by any caller, and must remain side-effect-free even under repeated/rapid calls (debounced typing) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-3-07 | Tampering | POST /api/goal-settings/preview accidentally persisting candidate weights | medium | mitigate | Task 1's behavior tests explicitly assert that a subsequent GET /api/goal-settings is unchanged after a preview call with different weights — this is a required test case, not an incidental one |
| T-3-08 | Information Disclosure | Preview endpoint leaking another user's goals via a missing id_pengguna scope | high | mitigate | Reuses `fetch_and_rank_goals(current_user_id)`, the same already-IDOR-safe aggregation function `GET /api/goals` uses — no new query path is introduced |
| T-3-09 | Denial of Service | Debounced-but-unbounded preview calls on every keystroke burdening the backend | low | accept | 300ms client-side debounce bounds request frequency to a level consistent with normal form-typing load; no rate limiting exists anywhere else in this codebase either, so this is not a new risk tier relative to the rest of the app |

</threat_model>
