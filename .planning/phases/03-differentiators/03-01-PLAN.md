<objective>
Build the shared Gemini 2.5 Flash integration layer that Plans 03-05 (receipt scan), 03-06 (e-statement import), and 03-07 (AI insights) all depend on: a cached client factory, three timeout-wrapped extraction/generation functions, the three Pydantic response schemas that double as Gemini's structured-output contract, and the Wave 0 test scaffolding (`fake_gemini_response` fixture + `test_gemini_service.py`) required by 03-VALIDATION.md before any downstream plan can be verified.

This plan does NOT wire any FastAPI route or frontend page — it produces the tested, reusable service module that Wave 2 plans call into. Per RESEARCH.md Pitfall 2, the critical acceptance criterion is that every Gemini call goes through the SDK's **async** client (`client.aio.models.generate_content`) inside `asyncio.wait_for(...)`, never the blocking sync client — a sync call inside `async def` cannot be interrupted by `asyncio.wait_for` and silently defeats the 10s/15s hard-timeout requirements (SCAN-03, AIINS-03) that 03-05/03-06/03-07 build on top of this module.

Purpose: Give every Phase 3 AI-backed feature one tested, non-duplicated integration point instead of three independent (and likely inconsistent) Gemini call sites.
Output: `backend/core/gemini_client.py`, `backend/services/gemini_service.py`, three new Pydantic model files, a new `conftest.py` fixture, and a passing `test_gemini_service.py`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| FastAPI backend -> Google AI Studio Gemini API | Server-to-server HTTPS call carrying `AI_VISION_API_KEY`; request payload includes user-supplied file bytes (receipt image / PDF) forwarded to a third-party LLM |
| Gemini API response -> Pydantic response_schema parsing | Untrusted LLM output must not be trusted as pre-validated even though `response_schema` constrains shape |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-3-01 | Tampering | LLM prompt injection via malicious receipt/PDF content attempting to manipulate extraction output | medium | mitigate | `response_schema` (Pydantic) constrains Gemini's output shape at the API level regardless of prompt-injection attempts in the image/PDF content; `Field(gt=0)` on `nominal` in `ReceiptExtraction`/`StatementTransaction` additionally rejects a manipulated non-positive amount at the model layer before it ever reaches a router |
| T-3-02 | Information Disclosure | `AI_VISION_API_KEY` leakage via error messages | high | mitigate | `gemini_service.py`'s broad `except Exception: return None` never re-raises or logs the raw SDK exception into any value returned to a caller — callers (03-05/03-06/03-07 routers) construct their own generic Bahasa Indonesia fallback message, never echo the underlying exception text to the client |
| T-3-03 | Denial of Service | Blocking the FastAPI event loop via a mis-wired sync Gemini call, freezing all concurrent requests | high | mitigate | Task 2 verifies (via `asyncio.wait_for` timeout assertions) that only the SDK's async client (`client.aio.models.generate_content`) is used, never the sync client inside an `async def` — per RESEARCH.md Pitfall 2, this is the single most consequential implementation detail in this plan |
| T-3-SC | Tampering | npm/pip supply-chain risk of a new dependency (`google-genai`) | high | mitigate | Task 0 blocking `checkpoint:human-verify` confirms the package's PyPI listing links to the official `googleapis/python-genai` GitHub org before install, per the mandatory package-legitimacy protocol for `[SUS]`-flagged packages |

</threat_model>
