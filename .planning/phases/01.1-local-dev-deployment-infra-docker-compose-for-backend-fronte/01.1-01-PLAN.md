    `backend` service: `build.context: .` and `build.dockerfile: backend/Dockerfile` (NOT `build: ./backend` — that breaks the `backend.` import namespace per Pitfall 5). Publish port `8000:8000`. Set `env_file: ./backend/.env`. Mount `./backend:/app/backend` for hot reload. Command: `uvicorn backend.main:app --host 0.0.0.0 --port 8000 --reload`.

    `frontend` service: `build.context: ./apps/web` (Dockerfile lives directly inside `apps/web/`, no cross-package import concern here). Publish port `3000:3000`. Mount `./apps/web:/app`, then add a second, anonymous volume entry `/app/node_modules` immediately after it (per Pitfall 3 — this keeps the container's own `npm ci`-installed `node_modules`, preventing the host bind mount from shadowing it). Command: `npm run dev -- -H 0.0.0.0` (explicit host bind so the published port is reachable from outside the container, not just `next dev`'s default).

    Create `backend/Dockerfile`: base image `python:3.12-slim`, `WORKDIR /app`, `COPY backend/requirements.txt backend/requirements.txt`, `RUN pip install --no-cache-dir -r backend/requirements.txt`, `COPY backend/ backend/`, `CMD` running `uvicorn backend.main:app --host 0.0.0.0 --port 8000` (no `--reload` in the image default — the compose `command:` above overrides this for dev with `--reload`).

    Create `apps/web/Dockerfile`: base image `node:20-alpine` (matches the documented Node >=20 minimum, not the host's newer Node), `WORKDIR /app`, `COPY package.json package-lock.json ./`, `RUN npm ci`, `COPY . .`, `CMD` running `npm run dev`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Git history / committed files | `.env.example` (placeholders) is intentionally committed; real `.env`/`.env.local` files must never cross into git |
| Docker image layers | Files `COPY`-ed into an image layer persist forever, even after later "removal," if pushed to a registry |
| Developer laptop → Supabase cloud | Backend container's env vars are the only credential path to the hosted Supabase project (no local DB) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01.1-01 | Information Disclosure | apps/web/.gitignore, backend/.env.example, apps/web/.env.example | high | mitigate | The `!.env.example` negation only un-ignores the placeholder file; root `.gitignore`'s `.env`/`.env.local`/`.env.*.local` patterns and `apps/web/.gitignore`'s blanket `.env*` pattern continue to block every real secret-bearing file from ever being tracked |
| T-01.1-02 | Information Disclosure | backend/Dockerfile, apps/web/Dockerfile | high | mitigate | Neither Dockerfile ever `COPY`s a real `.env`/`.env.local` file or bakes an `ENV <secret>=<value>` instruction; real secrets flow in only via docker-compose's `env_file:` directive at container start, never into an image layer |
| T-01.1-04 | Elevation of Privilege | apps/web/.env.example | critical | mitigate | apps/web/.env.example contains only NEXT_PUBLIC_API_BASE_URL and NEXT_PUBLIC_USE_MOCK — SUPABASE_SERVICE_ROLE_KEY (which bypasses RLS) is never added to any apps/web env file, since apps/web has zero Supabase SDK usage |
| T-01.1-SC | Tampering | Docker base images (python:3.12-slim, node:20-alpine); pip/npm installs inside the images | low | accept | Official Docker Hub base images, not registry packages. All Python deps installed (fastapi, uvicorn, PyJWT, supabase, pydantic, python-multipart, python-dotenv) are pre-existing Phase 1 dependencies already re-audited in 01.1-RESEARCH.md's Package Legitimacy Audit — no new package is introduced by this phase, so no blocking human-verify checkpoint is needed |
</threat_model>
