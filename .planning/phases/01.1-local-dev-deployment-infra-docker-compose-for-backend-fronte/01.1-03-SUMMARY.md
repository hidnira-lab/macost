## Accomplishments
- Confirmed live Vercel deployment: `https://macost.vercel.app` (HTTP 200)
- Confirmed live Railway deployment: `https://macost-production.up.railway.app` (`/health` returns `{"status":"ok"}`) — required manually generating a public domain in Railway's dashboard (Settings -> Networking -> Generate Domain), since Railway doesn't auto-assign one
- Verified CORS: the live Vercel origin receives a matching `access-control-allow-origin` header from the Railway-deployed backend, proving Plan 01.1-02's `allow_origin_regex` works against the real domain
- Diagnosed and fixed a live bug: UptimeRobot's monitor reported `405 Not Allowed` because `/health` was `@app.get`-only and never registered `HEAD`; changed to `@app.api_route("/health", methods=["GET", "HEAD"])`
- Confirmed UptimeRobot monitor is active and shows "Up" after the fix redeployed via Railway's auto-deploy-on-push

## Files Created/Modified

- `render.yaml` - deleted in a prior session (`a29ed17`) when the backend deploy target moved from Render to Railway

## Decisions Made
- Backend deploy target is **Railway**, not Render, as originally planned — Render's card verification repeatedly failed; documented in `.claude/CLAUDE.md` and this plan's frontmatter
- Railway requires a manual "Generate Domain" step per service (no config-file equivalent to Render's `render.yaml` Blueprint needed for a basic Dockerfile deploy)
- `/health` must accept both `GET` and `HEAD` — established as a pattern for any future health-check routes

## Deviations from Plan
- Plan originally specified Render for the backend; actual execution used Railway (decided and committed in the prior paused session, `a29ed17`, before this session resumed)
- Plan's `<verify>` commands referenced `https://macost-api.onrender.com` — actual verification in this session used the real Railway URL and Vercel URL instead, since the plan's URLs were placeholders pending the real dashboard connect
- Added an unplanned fix: `/health` HEAD-method support, discovered only once UptimeRobot's live monitor exercised the endpoint

## Issues Encountered
UptimeRobot monitor initially reported `405 Not Allowed` because the FastAPI `/health` route only registered `GET`. Root-caused via direct `curl -X HEAD` vs `curl -X GET` comparison, fixed with `@app.api_route(methods=["GET","HEAD"])`, pushed to `main`, confirmed Railway redeployed and the monitor turned green.
