## Docker Compose scope

| Option | Description | Selected |
|--------|-------------|----------|
| Backend only | docker-compose.yml only runs backend FastAPI; frontend stays manual `npm run dev` | |
| Backend + Frontend both | docker-compose.yml runs both backend FastAPI and frontend Next.js dev server as containers | ✓ |

**User's choice:** Backend + Frontend both.
**Notes:** Supabase stays hosted, not containerized — backend connects to Supabase cloud via env vars.

---

## Vercel deployment mode

| Option | Description | Selected |
|--------|-------------|----------|
| Static export for both | One next.config.ts, one build (`output: 'export'`) used for both Tauri and Vercel | ✓ |
| Dynamic (SSR) for Vercel, static export for Tauri | Two separate build targets/configs | |

**User's choice:** Static export for both.
**Notes:** Keeps a single build pipeline; means apps/web must avoid SSR-only features going forward since the same static build feeds Tauri and Vercel.

## Env var & secrets handling

| Option | Description | Selected |
|--------|-------------|----------|
| .env.example + manual sharing | Commit .env.example placeholders, share real values via chat/DM | ✓ |
| Secrets manager (1Password/Doppler) | Dedicated secrets tooling for the team | |

**User's choice:** .env.example + manual sharing.
**Notes:** Matches the tight 10-day timeline and small team size (4 people).

## Deployment trigger

| Option | Description | Selected |
|--------|-------------|----------|
| Auto-deploy on push to main | Render & Vercel auto-deploy on every push/merge to main | ✓ |
| Manual trigger / staging first | Requires approval or separate staging environment | |

**User's choice:** Auto-deploy on push to main.
**Notes:** No staging environment for this milestone — team small enough to catch main breakage quickly.
