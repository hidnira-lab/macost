<decisions>
## Implementation Decisions

### Docker Compose scope
- **D-01:** `docker-compose.yml` runs BOTH backend (FastAPI) and frontend (Next.js dev server) as containers — one `docker compose up` starts everything.
- **D-02:** Supabase is NOT containerized — it stays fully hosted (cloud). Backend container connects to Supabase cloud via env vars (`SUPABASE_URL`, keys, JWT secret). No local Postgres container.

### Vercel deployment mode
- **D-03:** `apps/web` uses ONE build config for both targets — static export (`output: 'export'` in `next.config.ts`) is used for BOTH the Tauri desktop wrapper AND the Vercel deployment. Vercel serves the static export output; no SSR/dynamic Next.js features, no Next.js API routes on Vercel.
- **Implication:** any future feature work in `apps/web` must stay compatible with static export (no `getServerSideProps`, no dynamic route handlers) since the same build feeds both Tauri and Vercel.

### Env var & secrets handling
- **D-04:** Commit `backend/.env.example` and `apps/web/.env.example` with placeholder values (documenting every required var: `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`, `SUPABASE_ANON_KEY`, `JWT_SECRET`, `NEXT_PUBLIC_API_URL`, `USE_MOCK`, etc.) — actual `.env` files stay gitignored.
- **D-05:** Real secret values are shared manually by Hidayat (Day) to the other 3 team members (Fertika, Khayyira, Zarra) via chat/DM — no secrets manager (1Password/Doppler) for this milestone. Simple, matches the tight 10-day timeline and small team size.

### Deployment trigger
- **D-06:** Both Render (backend) and Vercel (frontend) auto-deploy on every push/merge to `main` — no manual approval step, no separate staging environment. Matches the tight timeline; the team is small enough that `main` breakage is caught quickly.

### Claude's Discretion
- Exact docker-compose service names, network configuration, volume mount strategy for hot-reload, and Dockerfile structure for backend/frontend containers.
- Render/Vercel-specific config file details (`render.yaml`, `vercel.json`) and build command specifics.
- Whether `apps/native` (Tauri) needs anything different from this phase, or is purely a downstream consumer of `apps/web`'s static export output — should not require changes to the Tauri desktop build path confirmed working in Phase 1.

### Supabase project scope (resolved 2026-07-02, post-research)
- **D-07:** Local dev and production (Render) share ONE Supabase project — same `SUPABASE_URL`/keys used in `backend/.env` locally and in Render's dashboard env vars. No separate dev/prod Supabase projects for this milestone.

### UptimeRobot keep-alive (resolved 2026-07-02, post-research)
- **D-08:** Configuring UptimeRobot (or equivalent) to ping the Render backend's `/health` endpoint IS in scope for this phase — include as a lightweight task (external service signup + point at `/health`). STATE.md already ties this to unblocking Phase 2 integration testing.

### Env var naming (resolved 2026-07-02, from RESEARCH.md — supersedes D-04's example names)
- **D-09:** `.env.example` files MUST use the exact var names the code actually reads, not the illustrative names in D-04: `SUPABASE_SERVICE_ROLE_KEY` (not `SUPABASE_SERVICE_KEY`), `SUPABASE_JWT_SECRET` (not `JWT_SECRET`), `NEXT_PUBLIC_API_BASE_URL` and `NEXT_PUBLIC_USE_MOCK` (not `NEXT_PUBLIC_API_URL`/`USE_MOCK`) — verified against `backend/core/supabase.py`, `backend/dependencies/auth.py`, `apps/web/lib/api/client.ts`. Full list and naming of any additional env vars beyond these remains Claude's Discretion (derive from `API_CONTRACT.md` and existing code).
