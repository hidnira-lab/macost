## Summary

This phase is infrastructure/config work, not new application logic — the risk is not "which library to pick" but "matching the exact contracts the existing code already assumes." Direct inspection of `backend/core/supabase.py`, `backend/dependencies/auth.py`, and `apps/web/lib/api/client.ts` turned up **three naming mismatches between CONTEXT.md's `.env.example` var list and what the code actually reads** — these must be corrected during planning, not left as-is, or the app will crash on `KeyError` at backend startup. See `## Env Var Name Corrections (Load-Bearing)` below.

Beyond env var accuracy, four infra patterns need to be right: (1) Docker Compose must use the modern `services:`-only syntax (no `version:` key — obsolete since Compose v2, 2022) with an anonymous volume trick to stop the frontend bind mount from clobbering `node_modules`; (2) the backend's `from backend.routers import ...` absolute-import style means both the Dockerfile `WORKDIR` and the Render start command must run `uvicorn backend.main:app` from the **repo root**, not from inside `backend/`; (3) `apps/web/.gitignore` currently has a blanket `.env*` pattern that will silently block `.env.example` from ever being committed — this directly contradicts locked decision D-04 and must be fixed with a negation pattern; (4) because `output: 'export'` bakes `NEXT_PUBLIC_*` vars into the JS bundle at **build time**, and because that same static bundle is fetched from the **browser** (not container-to-container), `NEXT_PUBLIC_API_BASE_URL` for local Docker dev must point to `http://localhost:8000`, never the Compose-internal service name `http://backend:8000`.

**Primary recommendation:** Use a two-service `docker-compose.yml` (backend + frontend, no local Postgres per D-02), fix the three env var name mismatches and the `apps/web/.gitignore` pattern before writing `.env.example`, add production CORS origins to `backend/main.py`, and wire `render.yaml` (backend) + Vercel dashboard root-directory config (frontend) as committed, git-tracked deployment blueprints so both platforms auto-deploy on push to `main` per D-06.

### Pattern 1: Docker Compose modern syntax (no `version:` key)
**What:** Compose files no longer declare a `version:` top-level key; Compose v2 ignores it and prints an obsolescence warning.
**When to use:** Always, for any new `docker-compose.yml` written today.
**Example:**
```yaml
# Source: https://docs.docker.com/reference/compose-file/legacy-versions/ (Compose Specification)
services:
  backend:
    build: ./backend
    ports:
      - "8000:8000"
    env_file:
      - ./backend/.env
    volumes:
      - ./backend:/app/backend
    command: uvicorn backend.main:app --host 0.0.0.0 --port 8000 --reload

  frontend:
    build: ./apps/web
    ports:
      - "3000:3000"
    env_file:
      - ./apps/web/.env.local
    volumes:
      - ./apps/web:/app
      - /app/node_modules   # anonymous volume — prevents host bind mount from
                              # shadowing the container's installed node_modules
    command: npm run dev
```

### Pattern 2: Backend Dockerfile respects the `backend.` import namespace
**What:** `backend/main.py` uses `from backend.routers import auth, wallets` — an absolute import assuming the package `backend` is importable from the current working directory. The Dockerfile build context and `WORKDIR` must be the **repo root**, not `backend/`.
**When to use:** Any Dockerfile or process manager invocation of this backend, local or prod.
**Example:**
```dockerfile
# backend/Dockerfile — but build context is the REPO ROOT (see docker-compose.yml `build: ./backend`... note below)
FROM python:3.12-slim
WORKDIR /app
COPY backend/requirements.txt backend/requirements.txt
RUN pip install --no-cache-dir -r backend/requirements.txt
COPY backend/ backend/
CMD ["uvicorn", "backend.main:app", "--host", "0.0.0.0", "--port", "8000"]
```
**Important:** if `docker-compose.yml` sets `build: ./backend` (context = `backend/` dir), the Dockerfile's `COPY` paths above (`backend/requirements.txt`) won't resolve — the build context must be the repo root (`build: { context: ., dockerfile: backend/Dockerfile }`) so `backend/` exists as a subdirectory inside the image, matching the import structure. This is a common source of `ModuleNotFoundError: No module named 'backend'` — call it out explicitly as a plan verification step.

### Pattern 3: Frontend Dockerfile runs the Next.js dev server, not a production build
**What:** Per D-01, the frontend container runs `next dev` (hot reload), not `next build && next start` — production serving is Vercel's job (D-03), not Docker's.
**When to use:** Local dev only. Never used for the prod deploy path.
**Example:**
```dockerfile
# apps/web/Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
CMD ["npm", "run", "dev"]
```

### Pattern 4: Render Blueprint (`render.yaml`) for the backend
**What:** Git-tracked Infrastructure-as-Code for the Render web service — auto-detected when the repo is connected to a new Render Blueprint.
**When to use:** Commit at repo root; secret env vars use `sync: false` so real values are entered once via the Render dashboard, never committed.
**Example:**
```yaml
# Source: https://render.com/docs/deploy-fastapi (adapted for this repo's backend.main:app import structure)
services:
  - type: web
    name: macost-api
    runtime: python
    plan: free
    buildCommand: pip install -r backend/requirements.txt
    startCommand: uvicorn backend.main:app --host 0.0.0.0 --port $PORT
    healthCheckPath: /health
    envVars:
      - key: PYTHON_VERSION
        value: 3.12.2
      - key: SUPABASE_URL
        sync: false
      - key: SUPABASE_SERVICE_ROLE_KEY
        sync: false
      - key: SUPABASE_ANON_KEY
        sync: false
      - key: SUPABASE_JWT_SECRET
        sync: false
```
Note the `healthCheckPath: /health` wires directly to the already-implemented `@app.get("/health")` in `backend/main.py` — no new endpoint needed.

### Anti-Patterns to Avoid
- **Hardcoding Render's `$PORT` as `8000` in the Dockerfile/render.yaml `CMD`:** Render assigns a dynamic port; the start command MUST read `$PORT`, not a fixed value — a hardcoded port makes the service unreachable even though the process itself starts fine.
- **Using `http://backend:8000` as `NEXT_PUBLIC_API_BASE_URL` in Docker Compose:** this Compose-internal DNS name only resolves *inside* the Compose network. Since `NEXT_PUBLIC_*` vars are inlined into client-side JS executed by the developer's browser (on the host, outside the Compose network), the browser cannot resolve `backend` as a hostname. Use `http://localhost:8000` (works because the backend's port is published to the host).
- **Baking real secrets into a committed `render.yaml`/`vercel.json`/Dockerfile:** always use `sync: false` (Render) / dashboard env var UI (Vercel) / `env_file:` at runtime (Compose) — never `ENV SUPABASE_...=<real value>` in a Dockerfile, which would bake the secret into an image layer permanently (recoverable even after later "removing" it).

## Common Pitfalls

### Pitfall 1: `apps/web/.gitignore` blocks the `.env.example` D-04 requires
**What goes wrong:** `apps/web/.gitignore` line 34 has a blanket `.env*` pattern. This matches `.env.example` too, so `git add apps/web/.env.example` will silently be ignored — the file never reaches the repo, and D-04 (commit `.env.example` with placeholders) silently fails.
**Why it happens:** The default Next.js-generated `.gitignore` uses a broad wildcard for "any dotenv-ish file," which is safe-by-default but too broad for this project's intent to commit example files.
**How to avoid:** Add a negation line directly after the `.env*` pattern: `!.env.example` (gitignore negation must come after the broader exclusion pattern, in the same or a later-processed file).
**Warning signs:** `git status` doesn't show the new `.env.example` as untracked after creating it inside `apps/web/`.

### Pitfall 2: `SUPABASE_JWT_SECRET` / `SUPABASE_SERVICE_ROLE_KEY` naming drift
**What goes wrong:** If `.env.example` (and real `.env`) use the names from CONTEXT.md's discussion (`JWT_SECRET`, `SUPABASE_SERVICE_KEY`) instead of what the code reads, the backend crashes on the first request touching auth with `KeyError: 'SUPABASE_JWT_SECRET'`.
**Why it happens:** CONTEXT.md's D-04 var list was written from discussion/memory, not a code grep — a normal and expected gap between discuss-phase and research-phase.
**How to avoid:** Use the corrected names from `## Env Var Name Corrections` table above, verified directly against `backend/core/supabase.py` and `backend/dependencies/auth.py`.
**Warning signs:** `KeyError` traceback naming an env var at backend startup or first authenticated request.

### Pitfall 3: Docker bind mount overwrites container `node_modules`
**What goes wrong:** Mounting `./apps/web:/app` without also excluding `node_modules` causes the host's (likely empty, or wrong-OS-built) `node_modules` to shadow the container's `npm ci`-installed one, producing `Module not found` errors or native-binding architecture mismatches (e.g. `@tauri-apps/plugin-store`'s native parts, if any get pulled transitively).
**Why it happens:** Bind mounts are directory-level overlays — anything under the mounted host path replaces the container path entirely, including subdirectories populated during the image build.
**How to avoid:** Add an anonymous volume for `node_modules` specifically (`- /app/node_modules` after the main bind mount) so Compose keeps the container's own copy for that subpath.
**Warning signs:** `next dev` fails immediately with missing-module errors that don't reproduce with a plain `npm run dev` on the host.

### Pitfall 4: `NEXT_PUBLIC_*` vars are frozen at build time, not read at runtime
**What goes wrong:** Because `output: 'export'` produces a purely static bundle, `NEXT_PUBLIC_API_BASE_URL` (and `NEXT_PUBLIC_USE_MOCK`) get inlined into the JS at `next build` time. Changing the env var in Vercel's dashboard *after* a build has zero effect until the next rebuild/redeploy is triggered.
**Why it happens:** This is standard Next.js static-export behavior, but it's counter-intuitive coming from a "runtime env var" mental model common in server apps.
**How to avoid:** Set the real `NEXT_PUBLIC_API_BASE_URL` (pointing at the Render prod URL, `https://macost-api.onrender.com`) in Vercel's dashboard **before** the first deploy triggers a build; any later change requires an explicit redeploy, not just a dashboard save.
**Warning signs:** Frontend on Vercel keeps calling `localhost:8000` (or an old URL) in production despite the env var showing the correct value in the dashboard.

### Pitfall 5: Docker build context vs. `backend.` import namespace mismatch
**What goes wrong:** If `docker-compose.yml`/Render sets the build context to `backend/` directly (instead of repo root with `dockerfile: backend/Dockerfile`), the resulting image has no `backend` package at its root — `uvicorn backend.main:app` fails with `ModuleNotFoundError: No module named 'backend'`.
**Why it happens:** `backend/main.py`'s `from backend.routers import auth, wallets` requires `backend` to be an importable top-level package relative to the process's CWD/PYTHONPATH — this only works if the container's working directory is the repo root, with `backend/` as a subdirectory inside it.
**How to avoid:** Set Docker build context to the repo root; set `WORKDIR /app` (repo-root-equivalent) in the Dockerfile; `COPY backend/ backend/` (not `COPY . .` from inside a `backend/`-scoped context). Same applies to Render's `rootDir` setting — leave it unset/default (repo root) so `buildCommand: pip install -r backend/requirements.txt` resolves correctly.
**Warning signs:** Container builds successfully but crashes immediately on `docker compose up` with an import error, or Render deploy fails at the `startCommand` step (not the build step).

### Pitfall 6: Backend CORS allow-list doesn't include the Vercel origin
**What goes wrong:** `backend/main.py`'s `CORSMiddleware` currently hardcodes `allow_origins=["tauri://localhost", "https://tauri.localhost", "http://localhost", "http://localhost:3000"]`. Once `apps/web` is live on Vercel (e.g. `https://macost.vercel.app` or a custom domain), every fetch from that origin will be blocked by the browser with a CORS error — the app will appear completely broken from the deployed frontend even though the backend itself is healthy.
**Why it happens:** This origin list was written in Phase 1, before any deployment target existed — it was never revisited when Vercel entered scope in this discussion.
**How to avoid:** Add the production Vercel URL (and ideally the Vercel preview-deployment pattern, e.g. via `allow_origin_regex=r"https://macost.*\.vercel\.app"`) to `backend/main.py`'s CORS config as part of this phase's implementation, not left for a later "oops it's broken" fix.
**Warning signs:** Browser console CORS errors (`No 'Access-Control-Allow-Origin' header`) after the first Vercel deploy, even though `curl`-ing the API directly works fine.

### Pitfall 7: Render free-tier cold starts break demo timing expectations
**What goes wrong:** Render's free tier spins down services after ~15 minutes of inactivity; the next request triggers a 30-60 second cold start. This directly conflicts with `ALLOC-01`'s "modal appears within ≤2 seconds" requirement and could embarrass the team during the July 9-10 demo or July 14 Expo if the backend hasn't been pinged recently.
**Why it happens:** Standard Render free-tier behavior, already flagged as an open blocker in `.planning/STATE.md` ("UptimeRobot keep-alive for Render backend must be active before Phase 2 integration testing").
**How to avoid:** Set up UptimeRobot (or equivalent) pinging `/health` on an interval under 15 minutes, as already decided in the prior session. This phase should either configure it directly or produce a clearly flagged follow-up todo — don't let it silently fall through.
**Warning signs:** First API call after a period of inactivity takes 30-60s; subsequent calls are fast.

### Pitfall 8: Committing a real secret into a Docker image layer
**What goes wrong:** A common Dockerfile mistake is `COPY .env .env` or `ENV SUPABASE_SERVICE_ROLE_KEY=<real value>` baked directly into the image. Even if a later commit removes the line, the secret persists forever in that image's layer history and in any registry it was pushed to.
**Why it happens:** Looks like it "just works" locally and is faster than wiring `env_file:`/dashboard env vars correctly.
**How to avoid:** Never `COPY` a real `.env` file into a Dockerfile; only `.env.example` (placeholders) may ever be copied, and only for documentation purposes, never as a runtime config source. Real secrets flow in via `env_file:` (Compose, local only) or the Render/Vercel dashboard env var UI (prod) — both keep secrets out of image layers and git history.
**Warning signs:** `docker history <image>` shows a layer referencing `.env` or an `ENV` instruction containing what looks like a real key/secret.

| Pattern | STRIDE | Standard Mitigation |
